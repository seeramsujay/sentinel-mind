# SentinelMind Frontend Walkthrough

This document outlines the technical architecture and core features of the SentinelMind frontend, a React-based tactical dashboard.

## 1. Architectural Overview

The frontend is built with **React**, **Vite**, and **TailwindCSS**. It follows a modular structure focused on real-time data visualization and secure orchestration.

- **Entry Point**: [main.jsx](file:///home/krishna/sentinel-mind/frontend/src/main.jsx) - Mounts the app and wraps it in `BrowserRouter` and `AuthProvider`.
- **Routing**: [App.jsx](file:///home/krishna/sentinel-mind/frontend/src/App.jsx) - Defines public (Login, Register) and private (Dashboard, Assets, Upload) routes using a `ProtectedRoute` component.

## 2. Authentication & "Proxy Mode"

A unique feature of this project is the **Hybrid Authentication System** implemented in [AuthContext.jsx](file:///home/krishna/sentinel-mind/frontend/src/context/AuthContext.jsx).

### Why Proxy Mode?
Firebase typically requires a Public Web API Key. To simplify the prototype and use the `service-account.json` directly from the backend, we implemented a proxy:

1.  **Frontend**: If `VITE_FIREBASE_API_KEY` is missing in [.env](file:///home/krishna/sentinel-mind/.env), [firebase.js](file:///home/krishna/sentinel-mind/frontend/src/firebase.js) defaults to "Proxy Mode".
2.  **AuthContext**: Instead of calling the Firebase JS SDK, it makes Fetch requests to the [auth_proxy.py](file:///home/krishna/sentinel-mind/backend/auth_proxy.py) (`/api/login`, `/api/verify`).
3.  **Token Management**: JWTs returned by the proxy are stored in `localStorage` and sent in the `Authorization` header for subsequent requests.

## 3. Real-time Dashboard Intelligence

The [Dashboard.jsx](file:///home/krishna/sentinel-mind/frontend/src/pages/Dashboard.jsx) is the central hub of the application.

### Data Sync Strategy
- **Firestore Listeners**: It uses `onSnapshot` to listen to the `emergencies` collection.
- **Defensive Rendering**: Since Firestore might be uninitialized in "Proxy Mode", the dashboard contains checks to prevent crashing, falling back to a "Static Demo Mode" with high-fidelity mock data when keys are missing.

### Key UI Modules
- **Tactical Map (Center)**: An SVG-based grid that visualizes agent locations and active incidents. It uses "pulsing" CSS animations for high-priority events.
- **Agent Feed (Left)**: A terminal-style log stream showing real-time system events (Dispatch, Triage, Conflicts).
- **RAG SitReps (Right)**: Displays intelligence reports generated by the backend's Retrieval-Augmented Generation (RAG) pipeline.
- **Fairness Audits**: Shows the status of bias detection and ethics checks performed by the AI Orchestrator on every dispatch.
- **HITL Override**: A specialized UI for "Human-In-The-Loop" approvals, highlighting incidents that the system cannot resolve autonomously.

## 4. State Management

- **Global State**: Managed via `AuthContext` for user session and authentication status.
- **Local State**: `useState` is used for component-specific data like logs, sitreps, and map coordinates.
- **Real-time State**: Managed directly through Firestore subscriptions, ensuring the UI stays in sync with the backend orchestrator and logistics daemons.

## 5. Security & Protection

- **ProtectedRoute**: Ensures that sensitive pages (Dashboard, Asset Tree) are only accessible to authenticated users.
- **JWT Verification**: The `AuthContext` verifies the token with the backend on every page refresh to ensure the session is still valid.
